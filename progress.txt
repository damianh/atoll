### 2026-04-02 ‚Äî Story 17: Implement layout system

**Completed by**: claude-opus-4.6
**Commit**: (pending) (branch: astro-fork)

**What was done**:
- Created `LayoutAttribute` in `src/Atoll.Core/Components/LayoutAttribute.cs` ‚Äî `[Layout(typeof(BaseLayout))]` attribute for page types; validates that the layout type implements `IAtollComponent`; supports inheritance (`Inherited = true`) so derived pages inherit their base class's layout
- Created `LayoutResolver` in `src/Atoll.Core/Components/LayoutResolver.cs` ‚Äî static utility class providing:
  - `GetLayoutType(Type)` ‚Äî reads `[Layout]` attribute to get layout type
  - `HasLayout(Type)` ‚Äî checks if a type has a layout attribute
  - `ResolveLayoutChain(Type)` ‚Äî resolves full layout chain (outermost-first) with circular reference detection
  - `WrapWithLayouts(Type, RenderFragment)` ‚Äî wraps page content with all layout layers (innermost first, then outer)
  - `WrapWithLayouts(Type, RenderFragment, props)` ‚Äî same but passes props to the innermost layout
- Created `tests/Atoll.Core.Tests/Components/LayoutTests.cs` with 31 integration tests covering:
  - LayoutAttribute construction, validation (null type, non-IAtollComponent type)
  - GetLayoutType for types with and without layout
  - HasLayout for types with and without layout
  - ResolveLayoutChain: empty, single, nested (2-level), three-level, circular reference detection
  - WrapWithLayouts: no layout passthrough, single layout wrapping, nested 2-level chaining, 3-level chaining
  - Props passing to innermost layout
  - Null argument validation for all public methods
  - Integration with PageRenderer (end-to-end layout + DOCTYPE + head injection)
  - Layout with named slots and fallback content
  - Layout with async page content
  - HasSlot verification (default slot present, no extra slots)
  - Empty page content in layout
  - Layout attribute inheritance from base class
  - Layout attribute override in derived class

**Design decisions**:
- LayoutResolver is a static class (not instance-based) ‚Äî layouts are resolved purely from type metadata at render time
- Layout chain is outermost-first but built inside-out: innermost layout wraps page content, then each outer layout wraps the result
- Circular layout references are detected by tracking visited types in a HashSet
- Props overload passes props only to the innermost layout; outer layouts get empty props

**Build verification**: `dotnet build -c Release` ‚Äî 0 warnings, 0 errors
**Test results**: All 521 tests pass (162 routing + 359 core, 31 new layout tests)

- Created `tests/Atoll.Server.Tests/MiddlewareIntegrationTests.cs` with 28 integration tests covering:
  - Page rendering (basic, parameterized, catch-all, rest params)
  - DOCTYPE auto-prepend on page responses
  - Layout wrapping via `[Layout]` attribute (full chain: nav, slot, footer)
  - Endpoint dispatching (GET ‚Üí JSON, POST ‚Üí 201 Created)
  - 405 Method Not Allowed with Allow header for unsupported methods
  - Route params passed to endpoints
  - Text, redirect, and empty endpoint responses
  - 404 fallthrough when no route matches
  - Correct content types (text/html, application/json, text/plain)
  - Base path stripping and non-matching base path rejection

  - Multiple routes in a single app

**Key discoveries**:
- ASP.NET Core's `HttpResponseMessage` classifies the `Allow` header as a content header (`response.Content.Headers`), not a response header ‚Äî important for testing 405 responses
- `PageRenderer` does not have built-in layout awareness ‚Äî layout wrapping must be done by the caller using `LayoutResolver.WrapWithLayouts()` before passing to `PageRenderer`
- The request handler uses `ComponentDelegate` pattern to compose layout-wrapped rendering through `PageRenderer.RenderPageAsync(ComponentDelegate)`, getting both layout wrapping and DOCTYPE/head injection
- Test SDK was changed to `Microsoft.NET.Sdk.Web` to enable `WebApplicationFactory`/`TestServer` support
